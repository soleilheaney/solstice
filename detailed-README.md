
# Solstice – Detailed Technical Overview

**Solstice** is envisioned as a platform that **streamlines membership and event management for organizations and communities**. This document provides a comprehensive look at the platform’s purpose, the rationale behind its technical stack, and deep dives into how various parts of the system work. It also outlines future plans to extend Solstice with additional capabilities.

## Vision and Purpose

Solstice aims to empower clubs, teams, and organizations to **manage their members and events in one unified system**. The core idea is to reduce the friction in organizing community activities by providing features such as membership rosters, team collaboration tools, event scheduling, and participant engagement, all under a cohesive experience.

- **Membership Management:** Solstice will allow users to sign up, join organizations (teams), and manage their membership profiles. For organization admins, it provides tools to handle invitations, roles, and member directories, making it easy to onboard and oversee members.
- **Team Collaboration:** Teams (or organizations) are a first-class concept. A team could be a club, a company department, or any group with members. Solstice’s multi-tenant design means each team has its own space for events and membership, with appropriate access control. For example, a user can belong to multiple teams and switch context to manage different team events.
- **Event Scheduling:** At its heart, Solstice simplifies event creation and management. Teams can create events (meetings, meetups, classes, etc.), set details (time, location, description), and invite or allow members (or the public) to RSVP. The platform is built to handle recurring events, reminders, and any content an event might need (agendas, files, etc. in future iterations).
- **Seamless User Experience:** By leveraging modern web tech, Solstice provides fast page loads and interactive features so that managing an event or updating a membership profile feels snappy. We want minimal load times and a mobile-friendly design — so organizers and members can use it on the go.
- **Scalability and Extensibility:** As communities grow, Solstice is designed to grow with them. The underlying architecture can scale to many organizations and events, and new features (like payments or integrations with calendars and communication tools) can be added without a rewrite.

*Example Use Case:* Imagine a sports club using Solstice. The club admins create a team on Solstice and invite all their members to join via email (handled by the platform’s auth system). Members sign up and can see upcoming practice sessions or tournaments in the Events section. They RSVP to events, get automatic reminders, and can even see which team members are attending. Meanwhile, new members can request to join the club through Solstice’s interface, and admins approve them, all in one place. This scenario demonstrates Solstice’s goal to be a one-stop solution for community management.

## High-Level Architecture

Solstice’s architecture is built on a **serverless, full-stack React application model**. At a high level, it consists of:

- A **React front-end** that runs both on the server (for initial page load) and in the browser (for interactivity). This is powered by **TanStack Start**, which handles the universal rendering (SSR) and routing.
- A **serverless backend** running on AWS, facilitated by SST. This includes Lambda functions that render pages (and could also handle APIs), and an Amazon RDS Postgres database for persistence.
- **Authentication and Authorization** managed by Better Auth, which spans the frontend (for UI and hooks) and backend (for security, sessions, and database records).
- A **relational database (PostgreSQL)** accessed via Drizzle ORM. This stores all persistent data: user accounts, team memberships, event details, RSVPs, etc.
- **Infrastructure as Code** definition (via SST) that ties everything together, linking AWS resources (like the DB, and any storage or APIs needed) with the application code in a cohesive way.

**Why this stack?** The chosen technologies solve specific needs:
- *TanStack Start* provides a lean yet powerful framework for SSR without heavy abstraction. We wanted the benefits of SSR (SEO, fast initial paint) with the flexibility of React and client-side navigation. TanStack Start, being built on TanStack Router, gave us fine-grained control and innovative features like isomorphic data loaders and streaming responses ([TanStack Start: A New Framework Revolutionizing React Development | by Rahul Bhooteshwar | Learn With Rahul | Medium](https://medium.com/learnwithrahul/tanstack-start-a-new-framework-revolutionizing-react-development-4143de93fc7e#:~:text=What%20Are%20Isomorphic%20Loaders%3F)) ([TanStack Start: A New Framework Revolutionizing React Development | by Rahul Bhooteshwar | Learn With Rahul | Medium](https://medium.com/learnwithrahul/tanstack-start-a-new-framework-revolutionizing-react-development-4143de93fc7e#:~:text=TanStack%20Start%20is%20a%20React,fast%20and%20seamless%20user%20experiences)).
- *Serverless (SST on AWS)* gives us scalability out-of-the-box. We don’t maintain servers; our code runs in Lambdas that scale on demand. SST also simplifies local dev by emulating the cloud environment and lets us deploy consistently. It was a good fit to manage our infrastructure (like creating the database and linking environment variables) in code, alongside our application logic ([Drizzle with Amazon RDS and SST | SST](https://sst.dev/docs/start/aws/drizzle/#:~:text=Use%20Drizzle%20and%20SST%20to,your%20Amazon%20Postgres%20RDS%20database)).
- *Better Auth* was selected after evaluating authentication options. Rather than building auth from scratch or using a less integrated solution, Better Auth offers a full suite: social logins, email/password, session management, even 2FA and organization management. Crucially, it **owns our auth** – meaning user data stays in our database, and we have full control. It integrates with our stack (supports TanStack Start) and automatically works with Drizzle for storing credentials, which saves development time and ensures type safety in auth-related code ([Better Auth](https://www.better-auth.com/#:~:text=,api%2C%20well%20done%20%40better_auth)).
- *Drizzle ORM* was chosen for its type-safe approach to database operations. Every query is checked at compile time, reducing runtime errors. Drizzle’s migrations and schema definition are simple and align well with our TypeScript codebase. Also, using Drizzle means Better Auth can generate the user/session schemas in sync, and any additional tables we create (for events, teams, etc.) can be managed similarly.
- *Tailwind CSS & Shadcn/UI* address the UI consistency and speed of development. Tailwind’s utility classes let us style quickly without leaving our JSX, and its design system helps maintain consistency. Shadcn/UI provides a set of accessible, themeable components (built on Radix primitives) which means we get polished UI elements (dropdowns, dialogs, etc.) without significant custom effort. This choice ensures we don’t spend excess time on frontend groundwork and can focus on feature development.

Overall, the architecture follows a **modular monolith** approach: the front-end, back-end, and infra are tightly integrated but modular in code structure. We can build and test features end-to-end (from database to UI) in isolation, which is great for productivity.

## Technical Deep Dive

Let’s dive deeper into specific aspects of the stack to understand how Solstice works under the hood:

### Server-Side Rendering and Routing (TanStack Start)

Solstice uses **TanStack Start** to manage routing and rendering. TanStack Start extends TanStack Router with a server layer, enabling true SSR. This means when a user first hits a page (say the event list page), our serverless function will render the React components into HTML on the server and send that to the client. The benefits are twofold: faster first paint (the user sees content immediately) and better SEO (search engines can index rendered pages).

TanStack Start introduces the concept of **isomorphic loaders**, which allow data-fetching logic to be shared between server and client ([TanStack Start: A New Framework Revolutionizing React Development | by Rahul Bhooteshwar | Learn With Rahul | Medium](https://medium.com/learnwithrahul/tanstack-start-a-new-framework-revolutionizing-react-development-4143de93fc7e#:~:text=What%20Are%20Isomorphic%20Loaders%3F)). For example, if we have a loader that fetches upcoming events for the homepage, that loader will run on the server during SSR to pre-fetch events, but it can also run on the client (if navigating without a full page refresh) to fetch data for that page. This avoids duplicating API calls and keeps data logic centralized.

**Streaming SSR:** One standout feature we leverage is streaming SSR ([TanStack Start: A New Framework Revolutionizing React Development | by Rahul Bhooteshwar | Learn With Rahul | Medium](https://medium.com/learnwithrahul/tanstack-start-a-new-framework-revolutionizing-react-development-4143de93fc7e#:~:text=TanStack%20Start%20is%20a%20React,driven)). In cases where certain data is slower to load (imagine an external API call or a large database query), TanStack Start can send partial HTML to the client immediately, and stream the remaining content as it becomes ready. This way, the user might see most of a page loaded except a part (which might show a spinner) that fills in moments later. It prevents the “white screen” effect of waiting for everything. Streaming SSR is baked into how we write loaders and components – we mark what can be deferred.

**Routing:** The routing is defined using TanStack Router’s API. Instead of file-based routes (like Next.js), we explicitly configure routes, which gives us dynamic routing flexibility. We have routes for pages like `/login`, `/dashboard`, `/teams/:teamId`, `/events/:eventId`, etc., each pointing to React components and optionally associated loader logic. The router integrates with TanStack Query for caching, so transitioning between pages is as smooth as a single-page app when data is already fetched.

**Client-side hydration:** After the server sends the HTML, the React app hydrates on the client. Thanks to TanStack Start, this hydration is efficient – it can reuse the data fetched on the server (no double fetching) and our components become interactive quickly. We ensure any browser-specific code runs only on the client side (for example, date pickers or certain hooks that depend on `window`).

 ([Introducing TanStack Start – Frontend Masters Blog](https://frontendmasters.com/blog/introducing-tanstack-start/)) *Diagram: In a typical client-side rendered app (SPA), the initial page load sends almost no content, then the browser must download and execute scripts and fetch data, leading to a delay before the page fully displays ([Introducing TanStack Start – Frontend Masters Blog](https://frontendmasters.com/blog/introducing-tanstack-start/#:~:text=Image)). Solstice avoids this by using SSR.*  

 ([Introducing TanStack Start – Frontend Masters Blog](https://frontendmasters.com/blog/introducing-tanstack-start/)) *Diagram: With server-side rendering, Solstice sends a pre-rendered HTML page (`Full Page Displays`), so the user sees content immediately. The client JavaScript then loads and hydrates to make the page interactive ([Introducing TanStack Start – Frontend Masters Blog](https://frontendmasters.com/blog/introducing-tanstack-start/#:~:text=Image)).*  

In summary, TanStack Start gives Solstice a **performance edge and a great developer experience** – we write React components and data loaders in TypeScript, and the framework handles the rest (server rendering, caching, streaming). This setup is relatively new (TanStack Start is in beta), but we anticipate it will continue to mature. We’re staying updated on any breaking changes (see the TanStack Router issue tracker) and have structured our code in a way that isolation of TanStack-specific logic is possible if needed.

### Authentication and Authorization (Better Auth)

Authentication is central to Solstice (since nearly everything – memberships, teams, events – ties to user accounts). We use **Better Auth** to implement a robust auth system without starting from scratch.

**OAuth and Social Logins:** Out of the box, we configured Better Auth with **GitHub, Google, and Discord OAuth** providers. This means users can log in using any of those accounts. The OAuth flows are handled mostly by Better Auth’s library – for example, hitting “Sign in with Google” will redirect to Google, and after consent, redirect back to Solstice where Better Auth handles the token exchange. Because Better Auth is integrated with our framework, it knows how to handle these callbacks in the context of TanStack Start’s server. New OAuth providers can be added easily by updating the config (e.g., Twitter or others) ([react-tanstarter/src/lib/server/auth.ts at main · dotnize/react-tanstarter · GitHub](https://github.com/dotnize/react-tanstarter/blob/main/src/lib/server/auth.ts#L36#:~:text=socialProviders%3A%20)) ([react-tanstarter/src/lib/server/auth.ts at main · dotnize/react-tanstarter · GitHub](https://github.com/dotnize/react-tanstarter/blob/main/src/lib/server/auth.ts#L36#:~:text=google%3A%20)).

**Email/Password:** We have the option for classic email/password sign-up as well. Better Auth supports it (it’s just disabled by default in our config) ([react-tanstarter/src/lib/server/auth.ts at main · dotnize/react-tanstarter · GitHub](https://github.com/dotnize/react-tanstarter/blob/main/src/lib/server/auth.ts#L36#:~:text=%2F%2F%20https%3A%2F%2Fwww.better)). Enabling it would allow users to register with an email and password, which would be stored (securely hashed) in our database. We plan to enable this once we set up email deliverability (for confirmations, password resets). The advantage is some users prefer not to use social logins, so having both options covers all bases.

**Sessions and Security:** Upon successful login (via any method), Better Auth creates a session for the user. Sessions are stored in the database (using a `sessions` table via Drizzle) and a session cookie is set in the user’s browser. The session cookie is HTTP-only and contains a secure token that ties back to the session record. Because we have SSR, we rely on cookies to know who is logged in on each request. Better Auth’s session management includes automatic expiration and refreshing. We configured session caching for performance – e.g., caching session data in memory for quick verification ([react-tanstarter/src/lib/server/auth.ts at main · dotnize/react-tanstarter · GitHub](https://github.com/dotnize/react-tanstarter/blob/main/src/lib/server/auth.ts#L36#:~:text=session%3A%20)) (with a 5-minute TTL in our config). This reduces database lookups on every request while still keeping security (if a session is revoked, it will be cleared in short order).

**Multi-Tenancy – Organizations/Teams:** A key reason for choosing Better Auth was its support for multi-tenant user management ([Better Auth](https://www.better-auth.com/#:~:text=Multi%20Tenant)). Using the provided organization plugin, we can allow users to create and join organizations (teams). Under the hood, Better Auth will manage linking users to orgs (and roles within orgs). This saves us from designing that schema from scratch. In Solstice, we use this to represent “Teams”. For example, when a user creates a new team on our platform, Better Auth’s APIs help create an organization entry and assign that user as an admin. Inviting a user to a team can be done via Better Auth’s invitation system, which generates a unique invite link or code. When a new user accepts, Better Auth ties them into the organization seamlessly. We still control the UX, but the heavy lifting of data structure and validation is handled by the auth library.

**Authorization & Roles:** With teams in place, we also have roles (like admin vs member). Better Auth supports role-based access control by associating roles with organization membership. We define what an “admin” can do versus a regular member (for instance, only admins can create or delete events for a team, or manage memberships). These checks happen in our application logic (e.g., in a loader or action, we’ll verify the current user’s role before proceeding). Better Auth provides the data (user X is an admin of team Y), and we enforce it in code.

**Developer Experience:** Implementing Better Auth into TanStack Start was straightforward – we initialized it in our server entry (`auth.ts`) with a Drizzle adapter ([react-tanstarter/src/lib/server/auth.ts at main · dotnize/react-tanstarter · GitHub](https://github.com/dotnize/react-tanstarter/blob/main/src/lib/server/auth.ts#L36#:~:text=baseURL%3A%20process)) pointing to our database. The library generated the needed Drizzle schema for auth tables, which we integrated into our migrations. We use hooks or utilities provided by Better Auth on the client side to check if a user is logged in, to retrieve user profile info, etc. The **sign-in page** in our app uses Better Auth’s client to initiate OAuth flows. There is also a sign-out function that clears the session. Because all of this is strongly typed, we catch mistakes early (e.g., referencing an undefined field in the user profile).

In summary, Better Auth gives Solstice a **complete authentication solution**: third-party logins, traditional logins, session management, and team-based multi-tenancy – all integrated with our database and SSR environment. This allows us to focus on building features like profile management and invitations at a higher level, rather than dealing with low-level auth protocols.

### Database and ORM (PostgreSQL with Drizzle)

All of Solstice’s data lives in a **PostgreSQL** database. We access it through **Drizzle ORM**, which maps our TypeScript classes/definitions to SQL tables and queries. The database is the single source of truth for things like user accounts, team memberships, event details, and RSVP records.

**Schema Management:** We define our schema in code, using Drizzle’s schema definition functions (which feel similar to Prisma or TypeORM for those familiar). For example, we have definitions for a `users` table, `sessions` table (from Better Auth), an `organizations` and `organization_members` table (for teams and their members), an `events` table, etc. Each table schema is a TypeScript object that Drizzle uses to ensure queries use correct column names and types. Whenever we modify the schema, we use **Drizzle Kit** to generate a migration – this produces a SQL migration file that SST can run on the RDS database to apply changes. The project likely has a `drizzle.config.json` pointing to our database and migrations folder, so running `pnpm db generate` creates new migration files, and `pnpm db push` runs them.

**Type Safety and Queries:** Using Drizzle means when we write a query, e.g., to get all events for a team, the result types are inferred (no need for manual casting). For instance, a query like `db.select().from(events).where(eq(events.teamId, someId))` will return a promise of an array of `Event` type objects. This reduces runtime errors and helps with IntelliSense when writing complex queries or joins. We also appreciate that Drizzle allows using raw SQL when needed, but so far, the DSL has been sufficient.

**Better Auth Integration:** As noted, Better Auth provided the schema for auth-related tables. This includes `users`, which store basic profile info and auth identifiers, `sessions` for login sessions, possibly `accounts` (OAuth accounts), and `organizations` & `organization_members` if using the org plugin. Because these are generated via Better Auth, we didn’t have to manually write them – but we still treat them like the rest of our schema in Drizzle. If we needed to extend the user profile (say add a bio or avatar URL), we could modify the generated schema to include those fields and regenerate – Better Auth is flexible in that regard.

**Data Access Patterns:** In Solstice, most data access is done within TanStack Start loaders or action functions:
- When rendering a page (SSR), a loader might use Drizzle to fetch the necessary data. For example, the Events List page’s loader will query the `events` table for events visible to the current user (maybe all events in their team or public events).
- For mutations (like creating an event), we might have an action or use a form that on submission calls a server function. That function would use Drizzle to `INSERT` a new row into the `events` table. Drizzle returns the inserted object (or we query it afterward) which we can then use to update the UI optimistically via TanStack Query or by redirecting to the new event page.
- Because the app is serverless, each invocation gets a fresh connection to the database (managed by the Node PostgreSQL pool in Drizzle). SST helps here by possibly reusing the DB connection across invocations if warm (and in local `sst dev` it’s persistent).

**Transactions and Safety:** Drizzle supports transactions; if an operation involves multiple queries (e.g., create event and also create a default RSVP for the creator, hypothetical), we can wrap them in a transaction to ensure consistency. Also, since multiple Lambda instances could run in parallel, we rely on the database to maintain consistency (unique constraints, etc., to avoid duplicate entries).

**Database Deployment:** SST provisions the Postgres database in AWS (likely an RDS instance or Aurora Serverless). During local development, we have the option to connect to a local Dockerized Postgres or use a dev RDS in the cloud. The `.env` contains the `DATABASE_URL` that Drizzle uses. If using a cloud dev DB, AWS SSO (step in Getting Started) ensures we have network access (possibly via SST’s port forwarding). In production, SST will handle deploying the database and running migrations as part of deployment (or we run them manually via the SST console/CLI).

### Serverless Infrastructure (AWS via SST)

Solstice runs on a **serverless infrastructure** managed by SST (Serverless Stack). This means our application doesn’t run on a traditional server 24/7, but rather as cloud functions that execute on demand, scaling automatically. Here’s how SST and AWS come into play:

- **SST App and Stacks:** Our project defines an SST configuration (likely in `sst.config.ts` or similar) which declares the resources needed. At minimum, we use the `TanStackStart` construct provided by SST ([TanStackStart | SST](https://sst.dev/docs/component/aws/tan-stack-start#:~:text=The%20,TanStack%20Start%20app%20to%20AWS)) to deploy our front-end. This essentially tells SST: “take our TanStack Start app, deploy it to AWS Lambda and CloudFront.” SST handles wiring up a CDN (CloudFront) to serve static assets and edge locations, and a Lambda for SSR in given regions ([TanStackStart | SST](https://sst.dev/docs/component/aws/tan-stack-start#:~:text=match%20at%20L1313%20The%20regions,based%20on%20the%20user%E2%80%99s%20location)). We likely specified a custom domain or will in the future, which SST can manage (Route53 integration).
- **Database (RDS):** SST can create a **PostgreSQL RDS** database instance as part of the stack ([Drizzle with Amazon RDS and SST | SST](https://sst.dev/docs/start/aws/drizzle/#:~:text=2)). It sets up the necessary VPC and security group so that our Lambdas can connect to the DB securely. The connection string is stored as a secret and provided to our app as an environment variable. This way, we didn’t have to manually create the DB through the AWS console; it’s defined in code. For local development, SST might spin up a lightweight local proxy or allow direct connection to the remote DB.
- **Lambda Functions:** The SSR rendering, as well as any API routes or server-only functions, run in AWS Lambda. With TanStack Start, the entire app (including React SSR logic) is essentially one Node.js server application packaged for Lambda. SST ensures this lambda has the right permissions (e.g., to read from Secrets Manager for env vars, or to access the RDS database in the VPC) ([TanStackStart | SST](https://sst.dev/docs/component/aws/tan-stack-start#:~:text=Link%20resources%20to%20your%20TanStack,This%20will)). One cool thing SST does in dev is run the lambda code locally while tunneling resources, so `npx sst dev` runs our SSR code on our machine but mimics the cloud environment.
- **Static Assets and CDN:** The frontend build (JavaScript, CSS, etc.) is deployed to S3 or directly to CloudFront by SST. CloudFront then serves these files (with correct caching headers). When a user visits our site, CloudFront also routes dynamic requests to the Lambda. Because the Lambda is region-specific (we could deploy it to multiple regions for latency), CloudFront takes care of routing the user to the nearest region and caching content where possible.
- **API Routes & Functions:** While TanStack Start primarily focuses on SSR and page loads, we can also create additional serverless API routes if needed (for example, a webhook endpoint for payments or a real-time function). SST lets us define standalone Functions. For now, most of our needs are handled by loaders and actions within the SSR framework, but we keep in mind we can extend with API routes easily (perhaps using SST’s ApiGateway constructs).
- **Environment Management:** SST differentiates between dev/staging/production. We can have different settings or resources per stage. For example, in a staging environment, we might use a smaller DB instance or a different domain. The config file and CLI make it easy to deploy to a test stage vs production. Also, secrets (like API keys) are managed via `.env` for local and via SST’s mechanisms (like using AWS Parameter Store or Secrets Manager in production).
- **Logging and Monitoring:** By default, SST will forward Lambda logs to Amazon CloudWatch. During development (`sst dev`), these logs stream live to our console, which is helpful for debugging SSR or backend issues. We plan to integrate further monitoring (like using SST’s Live Lambda debug or adding an observability tool) as the project grows, to catch performance issues or errors in the wild.
- **Costs and Scalability:** Serverless means we pay only for what we use. If Solstice has intermittent usage or is just starting out with few users, our AWS costs remain low. As usage increases, AWS will seamlessly scale up concurrency (more Lambda invocations, handling more requests). RDS is the one constantly running piece; we chose an instance size that balances cost with expected load (and can upgrade as needed). In future, we might investigate Aurora Serverless v2 for the DB to scale that horizontally as well.

In essence, SST lets us develop **locally with cloud fidelity** and deploy **infrastructure reliably**. The infrastructure choices (Lambda, RDS, CloudFront) give Solstice global reach and high availability without a lot of DevOps overhead.

### UI Layer and User Experience

While the underlying architecture ensures performance and scalability, the user experience is defined by our **UI layer** – primarily React components styled with Tailwind and enhanced by the Shadcn/UI library.

- **Component Organization:** We maintain a set of reusable components in `src/components`. These include common UI elements like buttons, form inputs, modals, etc., often imported from or inspired by Shadcn/UI (which itself is a collection of Radix UI components styled with Tailwind). Using these ensures accessibility (keyboard navigation, screen reader friendly) and consistency in look and feel. For example, all our forms use a common Input component rather than `<input>` tags directly, which means they all look uniform and have validation styling built-in.
- **Styling and Theming:** Tailwind CSS provides us with utility classes to rapidly build layouts. We adhere to a consistent spacing, color, and typography scale defined by Tailwind’s config (extended with our brand colors perhaps). We’ve enabled dark mode support – our ThemeToggle component allows users to switch between light and dark themes, and we persist their preference. Because Tailwind v4 has JIT mode and other optimizations, our CSS bundle remains small, only including the classes we use.
- **State Management:** For most part, we rely on React’s built-in state and TanStack Query (for server state) to manage data. We haven’t introduced a global state library like Redux or Zustand because it hasn’t been necessary – TanStack Query and context APIs suffice. For instance, user authentication state is made available via context (Better Auth likely provides a hook or we wrote one using React context) so any component can know if the user is logged in and who they are. Similarly, we might use context or TanStack Query’s caching to share data like the current team info across components.
- **SSR and CSR Blend:** Because we use SSR, some interactive parts of the UI are carefully designed to not break when rendered on the server. We avoid directly using browser-only APIs in top-level component logic. If needed, we use effect hooks to do things after mount. This ensures that server-rendered HTML matches what the client expects. One example is forms: if a form has client-side validation, we ensure it also has a server-handling path in case of SSR submission. In general, TanStack Start encourages patterns that work both server and client, which we follow.
- **Forms and Validation:** We often use framework-agnostic solutions for forms (like React Hook Form or just controlled components) along with zod or Yup for schema validation. This could be an area of future improvement, but for now, basic form validation is in place (e.g., check required fields, format of email, etc.). Error messages are displayed using our form components styled via Tailwind.
- **Navigation and Feedback:** Thanks to TanStack Router, navigating between pages doesn’t reload the whole site – it’s instantaneous (client-side transitions with data fetched via loaders). We also provide user feedback for actions: e.g., a loading spinner on a button when submitting a form, toast notifications for success or error (possibly using a small library or a custom `<Toast>` component). These touches are important for a smooth UX, especially in a web app where users perform critical actions like creating events or inviting members.
- **Responsiveness:** The layout and components are designed mobile-first. Tailwind makes it straightforward to adjust styles at different breakpoints. We’ve tested key pages (dashboard, event detail, etc.) on mobile screen widths to ensure usability. The nav menus collapse into a mobile menu, and tables or lists (like member lists) can scroll or stack appropriately on smaller screens.
- **Accessibility:** Using Shadcn/UI (which wraps Radix) gives us accessible primitives (for dialogs, dropdowns, etc.). We also make sure to use proper HTML elements (e.g., `<button>` for clickable buttons, anchor tags for links, form labels connected to inputs). We run linting for accessibility where possible and manually verify using screen reader testing for critical flows. This is ongoing work, but a priority to make Solstice usable by all people.

Going forward, the UI will evolve as we get feedback. We might introduce more dynamic features (like drag-and-drop for reordering things, or live updates via websockets for real-time notification of RSVP counts, etc.), but the foundation with React/Tailwind is well-suited to accommodate these.

## Future Extensibility and Planned Features

Solstice is in active development, and we have a roadmap of features and enhancements. The chosen stack ensures we can add these with minimal friction. Here are some key future plans and how the stack will support them:

- **Payment Processing:** A likely near-future addition is handling payments – for example, paid membership subscriptions or event ticketing. We plan to integrate a payment processor for this but have not decided on one.
- **Calendaring and Integrations:** We want to make events easily accessible. In the future, we might add calendar integration (iCal feeds or Google Calendar sync). The stack can handle this by generating iCal files on demand (a serverless function that queries events and returns a calendar file) or by scheduling background jobs (SST could use EventBridge or a similar scheduled Lambda for periodic tasks, e.g., syncing with an external API). Also, integration with communication tools (like Slack notifications when an event is created, or email notifications to members) are on the table. AWS services (SNS or SES) could be utilized via SST to send emails or messages.
- **Real-time Features:** Right now, users need to refresh to see changes (or our loaders revalidate when navigating). In the future, we may add real-time updates – for example, see who else is currently viewing an event, or live updates on RSVP count. We could use WebSockets or server-sent events for this. SST supports WebSocket API routes, or we could integrate something like Ably or Pusher. TanStack Router would allow us to keep the UI in sync, and TanStack Query’s subscriptions could help update cached data. Our choice of React makes it possible to integrate these without a full rewrite.
- **Improved Admin Tools:** As the platform grows, we plan to add an admin dashboard for organization administrators to get analytics (like member growth, event attendance stats). This might involve more complex queries (e.g., aggregate queries for stats) – Drizzle can handle many, or we could create SQL views. The SSR pages for admin dashboard can fetch these stats server-side so the admin gets a quick overview. We might also incorporate charts (using a library like Chart.js or a lightweight React chart library).
- **Scalability and Optimization:** We are aware TanStack Start is new; if any limitations arise, we are prepared to contribute or find workarounds. Should the need arise to serve a massive amount of traffic, we can leverage AWS features like edge functions (CloudFront Workers) via SST for certain static pages or caching strategies. Also, as the database load increases, we can read-replicate the Postgres or use a caching layer (Redis, which SST can add easily). 
- **Testing & Quality:** We intend to build a suite of tests. TanStack Start doesn’t restrict testing – we can use Jest and React Testing Library for components, and possibly integration tests for the SSR routes (running them in a Node env). As we add features like payments, having tests will be crucial. SST’s ability to deploy ephemeral environments could help in automated testing for PRs in CI.

- **AI Assistance (LLM features):** Given the interest in LLMs, we might explore features like an AI assistant for event content. For example, a feature where organizers can input a few keywords and an AI (using OpenAI API or similar) generates a draft event description or suggests icebreaker questions for events. This would be an optional helper feature. Our architecture can support this by calling external APIs from server functions. We’d keep such calls on the server (to protect API keys and not expose them to the client). If we go this route, we’ll clearly document the addition. This is a speculative idea aligned with our innovation mindset.

In conclusion, Solstice’s foundation is strong – the stack was chosen not just for initial development convenience but for long-term maintainability and scalability. **We can iterate quickly** (thanks to type safety and modern frameworks) and **adapt** as requirements change (thanks to the flexibility of our serverless and modular design). The detailed design and decisions documented here will also help onboard new contributors and inform future decision-making. We will continue to update this document as the architecture evolves.

## Tickets Organization and Development Roadmap

To manage development, the project uses a ticketing system documented in the repository’s `/tickets/` directory. Tickets are grouped by major feature areas or development phases for clarity. This organization helps maintain focus on related tasks and makes it easier for contributors to find context for the work they’re doing.

The tickets are organized into the following subfolders under `/tickets/`:

- **Foundation (`tickets/foundation/`):** Core setup and infrastructure tasks. This includes initial project setup, configuration of the tech stack, and foundational features that everything else relies on.
- **Membership (`tickets/membership/`):** Features related to user accounts and membership management. E.g., user profile pages, authentication flows, handling of user roles outside of teams, etc.
- **Teams (`tickets/teams/`):** Everything about organizations/teams. Creating teams, inviting members, managing team info, roles within teams.
- **Events (`tickets/events/`):** Features for event creation, editing, listing, and RSVP management.

Each ticket is a Markdown file (`.md`) with a clear title. Inside each ticket file, we maintain a consistent format:
- **Summary:** A brief description of the feature or task and what it entails.
- **Goals:** The intended outcomes or what we aim to achieve when this ticket is completed. Often a bullet list of user stories or high-level requirements.
- **Technical Scope:** What’s included (and sometimes what’s explicitly excluded) in implementing the ticket. This might detail specific sub-tasks, changes to data models, API endpoints to create, UI components to build, etc.
- **Relevant Code/Docs:** Links to parts of the codebase or documentation that are relevant. For example, if a ticket is about improving authentication, it might link to `src/lib/server/auth.ts` or an external doc on Better Auth.
- **Implementation Notes:** Any tips, constraints, or suggestions on how to implement. This could include decisions made, potential libraries to use, or pitfalls to avoid. If some partial work is already done, it might note that as well.

The naming convention for ticket files often includes a number or slug for sorting, but the title inside the file is human-readable. For example, `001-initial-setup.md` might have a title “Initial Project Setup”.
